## EIP summary

EIP-8141 introduces a new EIP-2718 transaction type (`FRAME_TX_TYPE = 0x06`) that decomposes a transaction into a sequence of **frames** — contract calls with a mode (`DEFAULT`, `VERIFY`, `SENDER`) and flags (approval scope, atomic batch). Validity and gas payment become abstract: VERIFY frames call a new `APPROVE(scope)` instruction to authorize execution and/or payment; SENDER frames execute the actual operation as the user. Adds **6 new opcodes** (APPROVE, TXPARAM, FRAMEDATALOAD, FRAMEDATACOPY, FRAMEPARAM, SIGPARAM), two built-in signature schemes (SECP256K1 + P256), default code for EOAs, a canonical paymaster with reserved-balance mempool rules, atomic batching, and an expiry verifier at a fixed system contract address (`EXPIRY_VERIFIER = 0x8141`).

Status: Draft, Core, requires 1559/2718/3607/4844/7623/7702. One-sentence change: Ethereum's most ambitious account-abstraction proposal yet — frames replace the monolithic "one transaction does one thing" model with a composable validate-pay-execute pipeline in the protocol, not in ERC-4337's off-chain bundler layer.

## Hypothesis

The frame composition model can be modeled as a Solidity router on EVVM with opcode-level introspection mocked as functions. The semantic claim — "verify → pay → execute is composable across multiple frames with atomic batching and pluggable signature schemes" — holds at the contract layer, even though gas accounting, mempool policy, and opcode immediacy are not modeled.

This experiment **only** covers the foundation. Three sibling sub-experiments add (1) the canonical paymaster pattern, (2) default-code-for-EOAs handling, and (3) P256 signature support, each as a thin addition on top of this router.

## Dependencies and mocks

### Required EIPs (per `requires:` field of EIP-8141)

- **1559, 2718, 3607, 4844, 7623** — satisfied by stock EVM. (3607 is interesting: EIP-8141 explicitly disables it for frame txs; documented but not modeled here since we work at the contract layer.)
- **7702** — mocked as a small `DelegationRegistry` consulted by FrameRouter before treating a sender as EOA-with-default-code.

### Implicit deps (the ones that explode)

This experiment has 6 implicit deps. The minimal skill's dependency table flags >5 mocks as a decomposition signal, which is why this experiment is **one of four** rather than a single attempt at all of 8141.

- **APPROVE opcode (0xaa)** → mock-as-function (`FrameRouter.approve`). Limitation: loses opcode-frame-exit atomicity.
- **TXPARAM, FRAMEDATALOAD, FRAMEDATACOPY, FRAMEPARAM, SIGPARAM opcodes** → mocks-as-functions. Limitation: ~700 gas per call vs. 2-3 gas for opcodes.
- **Canonical expiry verifier at 0x...8141** → `MockExpiryVerifier` (simulate). Limitation: regular address, not canonical.
- **P256 signature scheme** → `MockP256Verifier` (vendor) wrapping Daimo's p256-verifier. Limitation: ~340k gas vs. ~3-7k for the EIP-7951 precompile.

The remaining concerns — mempool policy, banned opcodes, paymaster reservation accounting, default-code-install semantics — are **deferred** (they live outside the EVM and would require node software changes). See `mocks[]` in `manifest.json` for the full strategy table.

## Contracts produced

### `contracts/FrameRouter.sol`

**Type:** New service extending `EvvmService` (`new-service`)

**What:** A Solidity contract that accepts an array of frames and signatures, validates the signatures, walks the frames dispatching by mode, tracks per-tx approval state (`payer`, `senderApproved`) via Solidity 0.8.28+ transient storage, and emits `Approved(scope)` events from the mock APPROVE entrypoint. Provides view functions that model the introspection opcodes (`txParam`, `frameDataLoad`, `frameParam`, `sigParam`).

**Why:** Shape B is correct because frame transactions are conceptually a service-level abstraction on top of EVVM's existing primitives. The new "frame transaction" router is a fresh service that calls Core through its existing public surface (`pay`, `caPay`, `validateAndConsumeNonce`) — Core itself doesn't need modification. The single non-obvious decision: using `transient` storage (EIP-1153) for the per-tx context (`payer`, `senderApproved`, `currentFrameIdx`). Transient storage resets per tx automatically, mirroring the EIP's per-tx scope; regular storage would require manual reset and complicate the read paths.

**EIP mapping:**
- §"Frame Transaction" §"Payload Encoding" — modeled by the `Frame[]` and `Signature[]` parameters of `executeFrameTx`
- §"Frame Modes" — DEFAULT/VERIFY/SENDER handled in `_executeFrame` via the `mode` switch
- §"Behavior" steps 1-5 (sig validation, payer init, frame execution loop, payer check, refund) — implemented in `executeFrameTx`
- §"APPROVE Instruction" — modeled by `approve()`; emits `Approved(scope)` for off-chain observers
- §"Introspection" (TXPARAM, FRAMEDATA*, FRAMEPARAM, SIGPARAM) — view functions in the same file
- §"Atomic Batching" — `_executeAtomicBatch` handles the consecutive-flag-set frame group with rollback semantics

**Limitations:**
- **Six opcodes are mocked as functions.** Reviewers should treat this as an integration test for the state machine, not a faithful gas-cost model.
- **Mempool policy not modeled.** The entire §"Mempool" section of the EIP (validation prefix recognition, banned opcodes, paymaster reservation, validation trace rules) lives in node software, not in the EVM. Scaffold-evvm cannot model it.
- **Default code for EOAs not modeled here.** That's a separate sub-experiment (`eip-8141-default-code`). FrameRouter assumes the sender either has deployed code or has a delegation entry in DelegationRegistry.
- **P256 signatures supported but slow.** ~340k gas per verify until EIP-7951 ships.
- **Atomic batching's `SENDER` snapshot semantics** are simulated with a try/catch + revert pattern; the EVM's actual storage snapshot/restore is not exposed in Solidity, so deeply-nested reentrant-revert semantics may not match a real client implementation.

### `contracts/MockExpiryVerifier.sol`

**Type:** Mock contract (`mock-contract`) — simulate strategy

**What:** A tiny Solidity contract whose Solidity behavior is bit-equivalent to EIP-8141's canonical 26-byte runtime bytecode at `EXPIRY_VERIFIER = 0x8141`. Accepts 8-byte big-endian expiry timestamp as calldata; reverts unless `block.timestamp <= expiry_timestamp`.

**Why:** The EIP specifies an exact bytecode; we provide a Solidity equivalent so the FrameRouter's expiry handling can be exercised end-to-end without a `vm.etch` dance during local testing. For production fidelity, a deploy script would `vm.etch` the canonical 26 bytes at `0x...8141` — that's an integration concern outside the contract scope.

**EIP mapping:** §"Expiry Verifier Frame" — including the canonical bytecode reference.

**Limitations:**
- Deployed at a regular address, not the canonical `0x...8141`. Behavior is bit-equivalent; address is not.
- Permits TIMESTAMP opcode use (the EIP says TIMESTAMP is otherwise banned in VERIFY frames; this contract is the documented exception).

### `contracts/MockP256Verifier.sol`

**Type:** Mock contract (`mock-contract`) — vendor strategy

**What:** Thin wrapper around Daimo's `P256` library. Exposes a `verify(hash, r, s, qx, qy) → bool` matching what FrameRouter expects when dispatching signature scheme `0x1` (P256). The signer address derivation `keccak256(qx || qy)[12:]` is also exposed.

**Why:** P256 is a real EVM-level concern (EIP-7951 precompile) that isn't shipped yet. Daimo's pure-Solidity implementation has been audited and is production-grade — vendor (not mock) because the cryptographic correctness matters even when the gas profile doesn't. When EIP-7951 ships, this file becomes a one-liner that delegates to the precompile call.

**EIP mapping:** §"Signature Schemes" — specifically the P256 row in the table, and `validate_signature` for the P256 branch.

**Limitations:**
- ~340k gas per verify, vs. ~3-7k for the precompile. Functional behavior is correct; gas profile is not. Adversarial scenarios about gas exhaustion would not be representative.
- Vendor pin: ensure the deployed Daimo p256-verifier matches the version this wrapper was tested against. Document the commit hash in `notes/`.

## Open questions

- **APPROVE's frame-exit semantics.** The real opcode terminates the frame on success AND updates approval state in one step. Our mock requires the calling contract to return after invoking `approve()`. Is this difference observable in a meaningful adversarial test? Researcher's call.
- **Atomic batch rollback exactness.** Solidity's try/catch + revert is close to but not identical to the EVM's snapshot/restore for the SENDER atomic-batch case. Worth investigating in a dedicated adversarial scenario.
- **DelegationRegistry shape.** We mock EIP-7702 delegation as a simple `mapping(address => address)`. The real 7702 spec has more nuance around authorization lists and signed delegations. Probably out of scope for the foundation experiment; could be a fifth sub-experiment.
