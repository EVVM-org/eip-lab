## EIP summary

EIP-8182 introduces a **shielded-pool system contract** at fixed address `0x...081820` for private ETH and ERC-20 transfers. The pool uses a **split-proof architecture**: a hard-fork-managed Groth16 BN254 pool proof (handles value conservation, nullifiers, Merkle membership) + a permissionless auth proof (each user picks ECDSA/passkey/whatever via the auth-policy registry). Three external functions: `transact` (private spend), `deposit` (public→shielded), `setAuthPolicy` (register auth method). Two Merkle trees: depth-32 append-only note commitment tree + depth-32 sparse mutable auth-policy registry. Poseidon2 hash throughout with explicit domain separators per EIP §3.1.

Status: Draft, Core, requires EIP-20. One-sentence change: the protocol-level shielded pool Ethereum has been waiting for, with split-proof modularity so the auth layer can evolve without hard forks.

## Hypothesis

The contract-layer state machine — nullifier uniqueness, intent replay protection, auth-policy rotation window, pluggable auth verifier dispatch via `staticcall` — holds under the EIP's invariants when the cryptographic substrates are admin-controlled mocks. The split-proof architecture's separation of concerns is faithful to the EIP; only the cryptographic strength is stubbed.

## Dependencies and mocks

### Required EIPs

- **EIP-20** — satisfied by standard library (OpenZeppelin `IERC20`). The deposit/withdraw paths use the standard `transferFrom`/`transfer` interfaces with EIP-8182 §5.4.1's careful returndata-shape handling.

### Mocks introduced (4)

- **MockPoolVerifier** — stubs Groth16 BN254 verification. Strategy: **mock** (admin-controlled). Limitation: **NO cryptographic security** — verdicts are admin-set. Acceptable only because this experiment proves the state machine, not the proof system.

- **MockPoseidon** — stubs Poseidon2 over BN254. Strategy: **mock** (keccak with matched domain separators). Limitation: **NOT a real Poseidon** — circuit-generated hashes will not match. Tree roots are keccak-based, not Poseidon-real. Acceptable for state-machine testing; NOT acceptable for any binding-strength claim.

- **MockECDSAAuthVerifier** — reference auth verifier. Strategy: **mock**. Limitation: not a ZK proof — the "auth proof" is just an ECDSA signature byteblob. Demonstrates the contract integration only.

- **MockDummyAuthVerifier** — always-returns-true. Strategy: **mock**. By design has no security; for adversarial testing only.

5 strategy choices total (1 standard-library, 4 mocks). Right at the EIPLab threshold — comfortable for a single experiment.

## Contracts produced

### `contracts/ShieldedPool.sol`

**Type:** New service extending `EvvmService` (`new-service`)

**What:** The canonical system contract per EIP §5. Implements `transact`, `deposit`, plus the 13-step `transact` execution order (§5.4.1): chain-id verification, intent expiry, root-history acceptance for both trees, nullifier uniqueness, public-input range checks, pool-proof verification via MockPoolVerifier, auth-proof dispatch via `staticcall` to the user-supplied `authVerifier`, nullifier marking, intent-replay-ID consumption, output-note-data-hash verification, public asset movement (ETH or ERC-20), leaf index assignment, and event emission. Storage holds the note commitment tree (depth-32 append-only), the note commitment root history (circular buffer of size 500), the nullifier set, and the intent replay ID set.

**Why:** Shape B is correct because EIP-8182 §1 explicitly says "this EIP introduces no new precompile, opcode, transaction type, or other change to the Ethereum protocol" beyond installing the system contract. That makes the pool a contract-layer addition, which maps cleanly to a Shape B service. Building it as a new contract (rather than modifying Core) preserves EVVM's existing payment surface — the pool integrates via Core's existing `getPrincipalTokenAddress()` and the standard ERC-20 interfaces. In production this contract is installed at `SHIELDED_POOL_ADDRESS = 0x...081820` at fork activation; here it deploys normally and the deploy script would `vm.etch` for canonical fidelity.

**EIP mapping:**
- §5 (entire System Contract chapter) — this contract IS the system contract
- §5.4.1 (transact execution) — implemented step-by-step in `transact()` with comments marking each step
- §5.4.2 (deposit execution) — implemented in `deposit()`
- §5.3 (Contract Interface) — public function signatures match
- §5.5 (Pool Proof Verification) — dispatched via MockPoolVerifier
- §8 (Pool Circuit Requirements) — the contract enforces the public-input constraints the pool circuit also enforces (range checks, root acceptance) so the EIP's "if you fool the circuit, the contract still catches it" defense-in-depth claim holds

**Limitations:**
- **System contract not deployed at canonical address.** Regular address in this experiment; vm.etch at integration time.
- **Pool proof verification is mocked** — see MockPoolVerifier limitation. No cryptographic guarantee.
- **Tree storage uses a flat `mapping(uint256 => uint256)` keyed by leaf index.** Production implementations use packed leaf storage + incremental Merkle update for ~95% gas savings. The demo uses the simple layout for review clarity; the public surface is identical.
- **Reentrancy guards** are present but the `transact`/`deposit` paths are kept conservative (non-payable on transact, balance-delta check on deposit). Researchers should review the reentrancy surface against their own threat model.
- **Fee-on-transfer / rebasing tokens** are rejected per EIP §5.4.1 — the deposit balance-delta check catches fee-on-transfer; tokens that charge only on outbound `transfer` (not `transferFrom`) pass deposit but deliver less on withdrawal. The contract documents this gap.

### `contracts/AuthRegistry.sol`

**Type:** New service extending `EvvmService` (`new-service`)

**What:** Implements EIP §6 — the depth-32 sparse mutable Merkle tree keyed by `leafPosition`, the per-user `UserEntry` storage (`leafPosition`, `ownerNullifierKeyHash`, `noteSecretSeedHash`, `policySetCommitment`), the global `ownerNullifierKeyHashIndex` enforcing one-address-per-key-hash, and the block-based root history with window `AUTH_POLICY_ROOT_HISTORY_BLOCKS = 64`. Exposes `setAuthPolicy`, `getCurrentAuthPolicyRoot`, `isAcceptedAuthPolicyRoot`, and `getAuthPolicyEntry`.

**Why:** Split out of ShieldedPool for review clarity. The auth-policy registry is conceptually separate from note-spending — it manages user identity bindings and pluggable auth methods, which has its own lifecycle (rotation, deactivation, root-history aging). Keeping it in its own contract lets each be reasoned about independently, and lets the ShieldedPool call into it via a clean `IAuthRegistry` interface.

**EIP mapping:** §6 (entire Auth Policy Registry chapter), §3.4 (Merkle Tree Constructions, second paragraph on the auth-policy registry tree).

**Limitations:**
- **Same MockPoseidon caveat as ShieldedPool** — tree roots are keccak-based, not Poseidon-real. Circuit-generated hashes will not validate against these on-chain roots.
- **Block-based root history is implemented as a ring buffer of (root, blockNumber) pairs.** Window is exact (64 blocks); same-block updates correctly skip history entry creation per EIP §5.2.1.
- **Policy set tree is NOT maintained on-chain** per EIP §3.4 ("not maintained on-chain") — the wallet computes `policySetCommitment` off-chain and submits it; ShieldedPool's pool proof later verifies membership against this commitment.

### `contracts/MockPoolVerifier.sol`

**Type:** Mock contract (`mock-contract`) — mock strategy

**What:** Implements the `IMockPoolVerifier` interface that ShieldedPool dispatches to for pool-proof verification. Holds a `mapping(bytes32 publicInputsHash => bool verdict)` controlled by an admin. The `verifyProof(proof, publicInputs)` function returns the admin-set verdict for `keccak256(publicInputs)`; `proof` is unused. Also counts calls for sanity-check assertions.

**Why:** Real Groth16 verification requires a trusted-setup ceremony and costs ~250k gas per verify. For research scaffolding, neither is acceptable. The admin-controlled stub lets adversarial scenarios deterministically flip valid↔invalid without touching cryptographic infrastructure.

**EIP mapping:** §5.5 (Pool Proof Verification).

**Limitations:**
- **NO cryptographic security.** Acceptable only because this experiment proves the contract state machine, not the proof system.
- Verdicts are admin-set — production deployment of this would be a critical vulnerability.
- No public-input field-element validation (the real verifier rejects inputs `>= p` per EIP §3.5); we trust the inputs.

### `contracts/MockPoseidon.sol`

**Type:** Mock contract (`mock-contract`) — mock strategy

**What:** Solidity contract implementing the `IMockPoseidon` interface with a `poseidon(uint256[] memory inputs) → uint256` function. Internally uses `keccak256(abi.encode(inputs)) mod p` (BN254 scalar field order). Both ShieldedPool and AuthRegistry call this for all hash operations.

**Why:** Poseidon2 over BN254 is the EIP's specified hash function but it's not an EVM opcode. Real Solidity ports of Poseidon2 cost ~50k gas per hash, which makes the Merkle tree updates impractical for a research demo. Keccak with domain separators is collision-free and deterministic — sufficient for testing the contract state machine, even though it's not Poseidon-real.

**EIP mapping:** §3.3 (Poseidon Hash Construction), §3.1 (Domain Separators).

**Limitations:**
- **NOT a real Poseidon.** Circuit-generated hashes (from a real prover) will not match these on-chain hashes. Any roundtrip that involves generating a real proof off-chain and verifying it against this contract's tree will fail.
- The `_mod_p` operation truncates to the BN254 scalar field; collision properties of keccak-mod-p are weaker than real Poseidon-over-BN254 but adequate for state-machine testing.

### `contracts/MockECDSAAuthVerifier.sol`

**Type:** Mock contract (`mock-contract`) — mock strategy

**What:** A reference implementation of the `IAuthVerifier` interface that ShieldedPool dispatches to via `staticcall` per EIP §5.4.1 step 8. The auth proof is treated as an ECDSA signature; the function verifies it against an admin-registered signer for the given `authDataCommitment`. Demonstrates the contract integration for the pluggable-verifier pattern.

**Why:** Real auth verifiers wrap a ZK proof of credential possession (signature, passkey assertion, etc.). For the contract-layer experiment, demonstrating the dispatch path is the important part — a real ECDSA check is enough to make the wiring testable end-to-end without a ZK prover.

**EIP mapping:** §11 (pluggable auth verifier pattern), §8.1 step 8 (auth-proof public inputs `[blindedAuthCommitment, transactionIntentDigest]`).

**Limitations:**
- **Not a ZK proof.** No knowledge-of-secret guarantee. The "auth proof" is just an ECDSA signature byteblob.
- The relationship between `authDataCommitment` and the underlying credential is not cryptographically bound — the admin can register arbitrary signers for arbitrary commitments. Real auth circuits would prove this binding inside the ZK relation.

### `contracts/MockDummyAuthVerifier.sol`

**Type:** Mock contract (`mock-contract`) — mock strategy

**What:** Implements `IAuthVerifier` with a `verifyAuth(...) → bool` that always returns `true`. ~25 LOC total.

**Why:** Researchers writing adversarial scenarios can wire this in to confirm the pool's safety boundary doesn't depend on the auth circuit alone — i.e., that the pool proof AND the auth proof must both be valid for `transact` to succeed. Useful for proving the EIP's "split-proof" claim: if either proof is fake, the call still reverts because the other one catches it (assuming the pool proof is real, which in this experiment it isn't... so this mock is mostly useful for compositional testing alongside a real-circuit-replacement upgrade path).

**EIP mapping:** §11 (pluggable verifier registry).

**Limitations:**
- **By design has no security at all.** Production use of this would be a critical vulnerability. Existence in this folder is for negative testing only.

## Open questions

- **Merkle tree implementation.** The demo uses a flat mapping; production needs incremental Merkle update with packed leaf storage. The right place to do that work is a follow-up "production-grade" branch, not this contract-layer experiment.
- **Real verifier swap-in path.** When researchers replace MockPoolVerifier with a real Groth16 verifier, they'll need to coordinate the verification key with the circuit's trusted setup. Worth a separate writeup once the contract-layer experiment is settled.
- **Fee-on-transfer token handling.** EIP §5.4.1 documents the gap. Should the contract maintain an explicit allowlist of well-behaved tokens? Out of scope for this experiment; flagged for the EIP discussion.

## Handoff

The contracts in this folder and this justification are the deliverable. Researcher next steps (outside skill scope):

- **Write tests** with your own Foundry / Hardhat suite. Suggested adversarial cases: nullifier replay rejection, intent-replay-ID rejection, auth-policy root window expiration, pool-proof-fail + dummy-auth-pass combination still reverts, deposit balance-delta check rejects fee-on-transfer.
- **Deploy** in dependency order: MockPoseidon → MockPoolVerifier → AuthRegistry → ShieldedPool → (any auth verifiers researchers want to register).
- **Replace mocks one at a time** when graduating to real-circuit testing. Replace MockPoolVerifier first (with a Groth16 verifier generated from your circuit + trusted setup); then MockPoseidon (with a real Poseidon2 port if gas allows, or move to a hashing-via-precompile EIP); then iterate on the auth verifiers.
